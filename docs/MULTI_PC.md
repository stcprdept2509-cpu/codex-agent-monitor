# Multi-PC / Company Use

## 結論

他のPCや他社環境でも使えます。  
ただし、最初の実装形は「各PCでローカル起動」です。

理由は、Codex / Claude Code の実行履歴が各PCのローカルファイルに保存されるためです。  
クラウド上のWebアプリから、他人のPC内にある `~/.codex` や `~/.claude` を直接読むことはできません。

## 方式A: 各PCでローカル起動

一番安全で、すぐ使える方式です。

```bash
git clone <repository-url>
cd codex-agent-monitor
python3 server.py
```

表示:

```text
http://127.0.0.1:8799/
```

向いている用途:

- 自分のPCでAIエージェント稼働を撮影する
- 社内メンバーが各自のPCで見る
- 外部にデータを送らずに使いたい

## 方式B: LAN共有

同じWi-Fiや社内ネットワークから見たい場合は、起動時に `HOST=0.0.0.0` を指定します。

```bash
HOST=0.0.0.0 PORT=8799 python3 server.py
```

注意:

- 作業名やスキル名が見えるため、社外秘の情報が画面に出る可能性があります。
- 必要なときだけ使ってください。

## 方式C: 中央サーバー型

複数社・複数PCの状況を1つのWebアプリで見たい場合は、追加設計が必要です。

構成:

1. 各PCにローカルコレクターを置く
2. コレクターが最小限の状態だけ中央APIへ送る
3. 中央Webアプリが会社別・ユーザー別に表示する

必要な設計:

- ログイン認証
- 会社ごとのデータ分離
- 送信する項目の絞り込み
- 作業タイトルの匿名化
- 保存期間
- 退職者や外部委託先のアクセス停止

この方式はGitHub配布版の次の段階です。

## Vercelなどに置けるか

UIだけなら置けます。  
ただし、Vercel上のサーバーは利用者のPC内にある `~/.codex` / `~/.claude` を読めません。

そのため、Vercel公開だけで「他社PCのエージェント稼働状況が見える」状態にはなりません。

本当にクラウド型にする場合は、方式Cの中央サーバー型にしてください。
