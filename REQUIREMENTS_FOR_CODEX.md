# Agent Monitor — 要件定義(Codex引き継ぎ用)

デザイン・見た目については一切指定しない。ここに書くのは「何を」「どこから」「どう判定して」出すか、という中身の要件だけ。見た目・レイアウト・配色・アニメーションはCodex側で自由に決めてよい。

## 1. 目的

岩野が使っている **Codex** と **Claude Code**、両方の全エージェント(本体・サブエージェント問わず)が、今この瞬間どれだけ稼働しているかを、一目で分かるように見せるローカルアプリ。地味な数値一覧ではなく、実際に動いている実感が伝わる見せ方にすること。ただし表示する数値・状態はすべて実データに基づくこと。捏造した数値・雰囲気だけの演出は不可。

## 2. 対象データソースと取得方法

### Codex
- スレッド一覧: `~/.codex/state_5.sqlite` の `threads` テーブル(id, title, preview, agent_nickname, agent_role, model, created_at_ms, updated_at_ms, rollout_path, archived)。
- 親子関係(本体→サブエージェント): 同DBの `thread_spawn_edges` テーブル(parent_thread_id, child_thread_id)。
- 実行中判定・使用スキル: `rollout_path` が指すセッションログ(jsonl)を末尾から読む。`type:"event_msg"` の `payload.type` が `task_started` で、その後に `task_complete`/`turn_aborted` が来ていなければ「実行中」。使用スキルは `type:"response_item"` かつ `payload.name == "exec_command"` の `arguments.cmd` に含まれる `skills/<name>/SKILL.md` のパスから抽出する。
- **重要な注意点**: セッションログは長時間の会話だと数百MBになることがあり、末尾の固定サイズ(数十万バイト程度)だけ読むと、直近の`task_started`マーカーより前の部分しか読めず「実際は動いているのに待機中と誤判定する」バグが起きる。読み取り範囲は見つかるまで段階的に広げること(既存実装が対策済みなので参考にしてよい)。

### Claude Code
- セッション一覧: `~/.claude/projects/*/*.jsonl`(1ファイル=1セッション)。
- タイトル: エントリの `type:"custom-title"` の `customTitle`、無ければ `type:"ai-title"` の `aiTitle`、どちらも無ければ最初のユーザー発言。
- 実行中判定: `message.content` 内の `type:"tool_use"` ブロック(id, name, input)と、後続ユーザーメッセージ内の `type:"tool_result"` ブロック(`tool_use_id`)を突き合わせ、ファイル終端まで対応する `tool_result` が来ていない `tool_use` があれば「実行中」。
- 使用スキル: 実行中の `tool_use.name` が `"Skill"` の場合、`input.skill` がスキル名。
- サブエージェント: 実行中の `tool_use.name` が `"Agent"` の場合、`input.description` と `input.subagent_type` がそのサブエージェントの説明・種別。ただしこれはその瞬間だけ分かる一時的な情報で、Codexの親子関係のような永続レコードではない。

## 3. 表示すべき構造(組織図の考え方)

エージェントを以下の3階層で捉える。

1. **本体エージェント**(そのセッション/スレッドそのもの)
2. **サブエージェント**(本体から生まれた子。Codexは`thread_spawn_edges`、Claude Codeは実行中の`Agent`ツール呼び出しから分かる)
3. **使用中のスキル**(本体・サブエージェントそれぞれが今読み込んでいるスキル。上記2のデータソース参照)

この3階層はそれぞれ独立で良い(本体だけの場合、本体+スキルの場合、本体+サブエージェントの場合、本体+サブエージェント+サブエージェントのスキルの場合、などが起こり得る)。

## 4. 状態の種類

- **実行中**: 上記の判定ロジックで「まだ完了イベントが来ていない」もの。
- **待機中**: それ以外(完了済み、またはそもそも動いていない)。
- **エラー**: Codexで`turn_aborted`が最後の完了イベントだった場合。

## 5. 「種類」による分類

エージェントを「何の仕事をしているか」(SNS運用、営業、採用、経理、開発、など)で分類して見せること。CodexにもClaude Codeにも「職種」のような情報は存在しないため、以下の優先順位で推測する。

1. 使用中スキル名からの対応表(例: `stc-tsujimura-invoice-workflow` → 経理・請求書)
2. スキルが不明な場合は、タスク内容の文言に含まれるキーワードでの推測
3. どちらでも分からなければ「その他」

**この分類は推測であり、間違うことがある**。UI上でも「推測である」ことが伝わる扱いにするか、少なくとも誤分類を過度に断定しない見せ方にすること。

## 6. 表示範囲

直近24時間以内に動きがあったスレッド/セッションのみを対象とする(それより古いものは表示しなくてよい)。

## 7. データの正直さの原則

- 実データから計算できない数値(進捗率、CPU使用率など)は表示しない。表示するなら「実際に計算できる指標」に置き換える(例: 表示中のうち実行中の割合、実際の経過時間、実際のターン数など)。
- Codexの`Automation:`プレフィックスのような内部的な文字列はそのまま出さず、人が読んで分かる形に加工する。

## 8. 既存実装について

`server.py` に上記のデータ取得・判定ロジックがすべて実装済み(Python標準ライブラリのみ、`/api/state` がJSONで返す)。この裏側のロジックは動作確認済みなので、そのまま流用してデザイン部分(`index.html`)だけ作り直す進め方を推奨するが、必須ではない。全部作り直す場合も、このドキュメントの2〜7を満たすこと。

## 9. 完成の基準

- Codexのスレッドと Claude Codeのセッション、両方が同じ画面に出ること。
- 実際に動いているエージェントが、今CodexかClaude Codeで何か作業させれば、数秒以内に「実行中」として反映されること。
- 本体・サブエージェント・使用中スキルの3階層が、それぞれ実在する時だけ表示されること(捏造しない)。
- 「その他」に分類される割合や、分類の根拠が確認できること(隠さない)。
